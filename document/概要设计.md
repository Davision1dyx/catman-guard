# 智能运维Agent系统概要设计

## 1. 文档概述

### 1.1 编写目的
本文档为智能运维Agent系统的概要设计文档,旨在从技术层面描述系统的整体架构、模块划分、接口设计、数据库设计等内容,为后续的详细设计和开发实现提供指导。

### 1.2 项目背景
智能运维Agent系统旨在通过AI技术提升运维效率,构建包含智能问答、知识管理、运维统计等功能的综合性运维支持平台。

### 1.3 术语定义
- **LLM**: Large Language Model,大语言模型
- **RAG**: Retrieval Augmented Generation,检索增强生成
- **SSE**: Server-Sent Events,服务器推送事件
- **Vector Store**: 向量数据库
- **Embedding**: 文本向量化

## 2. 系统总体设计

### 2.1 设计原则

#### 2.1.1 高可用性
- 关键服务采用主备或集群部署
- 数据库主从分离,自动故障切换
- 服务熔断降级机制

#### 2.1.2 可扩展性
- 微服务架构,服务独立部署
- 数据库支持分库分表
- 向量数据库支持分片

#### 2.1.3 高性能
- 流式输出降低感知延迟
- 多级缓存提升响应速度
- 异步处理提升吞吐量

#### 2.1.4 安全性
- 多层安全防护
- 最小权限原则
- 敏感数据加密

public class DocumentReaderHandler {

#### 3.3.2 问题管理模块

**状态机设计**:
```java
public enum IssueStatus {
    NEW("新建"),
    PENDING("待处理"),
    PROCESSING("处理中"),
    RESOLVED("已解决"),
    CLOSED("已关闭");
}

public class IssueStateMachine {
    private static final Map<IssueStatus, Set<IssueStatus>> TRANSITIONS = Map.of(
        NEW, Set.of(PENDING),
        PENDING, Set.of(PROCESSING, PENDING), // 可转派
        PROCESSING, Set.of(RESOLVED, PENDING), // 可转派
        RESOLVED, Set.of(CLOSED, PROCESSING), // 可重新打开
        CLOSED, Set.of()
    );
    
    public boolean canTransition(IssueStatus from, IssueStatus to) {
        return TRANSITIONS.get(from).contains(to);
    }
}
```

-- 创建HNSW索引(高性能向量检索)
CREATE INDEX ON vector_store

@Service
public class CacheService {
    
    @Autowired
    private Cache<String, SessionData> sessionCache;
    
    @Autowired
    private Cache<Long, UserInfo> userInfoCache;
    
    /**
     * 获取会话数据
     */
    public SessionData getSession(String sessionId) {
        return sessionCache.get(sessionId, key -> {
            // 缓存未命中,从数据库加载
            return loadSessionFromDB(key);
        });
    }
    
    /**
     * 保存会话数据
     */
    public void saveSession(String sessionId, SessionData data) {
        sessionCache.put(sessionId, data);
        // 同时持久化到数据库
        saveSessionToDB(data);
    }
    
    /**
     * 获取用户信息
     */
    public UserInfo getUserInfo(Long userId) {
        return userInfoCache.get(userId, key -> {
            return loadUserInfoFromDB(key);
        });
    }
    
    /**
     * 清除用户缓存
     */
    public void clearUserCache(Long userId) {
        userInfoCache.invalidate(userId);
    }
}

#### 4.3.5 Caffeine优势

- ✅ **高性能**: 本地内存缓存,毫秒级响应
- ✅ **低延迟**: 无网络开销
- ✅ **简单**: 无需额外部署和运维
- ✅ **轻量**: 单机部署友好
- ✅ **功能丰富**: 支持TTL、LRU、统计等

#### 4.3.6 注意事项

- ⚠️ **单机缓存**: 不适合分布式场景的数据一致性要求高的缓存

- 💡 **建议**: 对于需要持久化或多节点共享的数据,应存储在数据库中

#### 4.4.3 文件存储服务

```java
@Service
public class FileStorageService {
    
    @Value("${file.storage.local.base-path}")
    private String basePath;
    
    /**
     * 保存文件
     */
    public String saveFile(MultipartFile file, String category) {
        // 生成文件ID和存储路径
        String fileId = UUID.randomUUID().toString();
        String fileName = file.getOriginalFilename();
        String filePath = buildFilePath(category, fileId, fileName);
        
        // 创建目录
        File destFile = new File(basePath + filePath);
        destFile.getParentFile().mkdirs();
        
        // 保存文件
        file.transferTo(destFile);
        
        return filePath;
    }
    
    /**
     * 构建文件路径
     */
    private String buildFilePath(String category, String fileId, String fileName) {
        // 按日期和分类组织目录: /uploads/2026/04/27/{category}/{fileId}_{fileName}
        LocalDate today = LocalDate.now();
        String datePath = today.format(DateTimeFormatter.ofPattern("yyyy/MM/dd"));
        return String.format("uploads/%s/%s/%s_%s", datePath, category, fileId, fileName);
    }
    
    /**
     * 读取文件
     */
    public InputStream readFile(String filePath) throws FileNotFoundException {
        File file = new File(basePath + filePath);
        return new FileInputStream(file);
    }
    
    /**
     * 删除文件
     */
    public boolean deleteFile(String filePath) {
        File file = new File(basePath + filePath);
        return file.delete();
    }
}

redisTemplate.opsForValue().set(cacheKey, answer, 1, TimeUnit.HOURS);
```

## 6. 安全设计

### 6.1 认证授权

#### 6.1.1 认证方案
- JWT Token认证
- Token有效期: 2小时
- Refresh Token有效期: 7天

**认证流程**:
```
用户登录
  ↓
验证用户名密码
  ↓
生成JWT Token和Refresh Token
  ↓
返回Token给客户端
  ↓
客户端请求携带Token
  ↓
服务端验证Token
  ↓
解析用户信息
```

**Token结构**:
```json
{
  "userId": "user-uuid",
  "username": "zhangsan",
  "role": "admin",
  "exp": 1682572800,
  "iat": 1682565600
}
```

#### 6.1.2 权限控制
- 基于角色的访问控制(RBAC)
- 权限粒度: 功能权限 + 数据权限

**角色定义**:
```java
public enum Role {
    OPS_STAFF("一线运维人员"),
    DUTY_STAFF("值班人员"),
    MANAGER("管理人员"),
    ADMIN("系统管理员");
}
```

**权限注解**:
```java
@PreAuthorize("hasRole('MANAGER')")
@PostMapping("/files/upload")
public Result uploadFile() {
    // 上传文件
}
```

### 6.2 数据安全

#### 6.2.1 敏感数据加密
- 密码: BCrypt加密
- 敏感配置: AES加密
- 传输加密: HTTPS

#### 6.2.2 SQL注入防护
- 使用MyBatis参数化查询
- 输入参数验证

```java
// 安全的查询
@Select("SELECT * FROM user WHERE username = #{username}")

#### 6.2.3 XSS防护
- 输出转义

### 6.3 接口安全

#### 6.3.1 限流

## 7. 性能优化设计

### 7.1 缓存优化

#### 7.1.1 多级缓存
```
请求 → Caffeine本地缓存 → PostgreSQL数据库
```

#### 7.1.2 缓存策略
- 热点数据: 预加载到Caffeine
- 查询结果: 短期缓存(1小时)
- 会话数据: 长期缓存(7天)
- 用户信息: 中期缓存(24小时)

**缓存优化建议**:
- 合理设置最大容量,避免OOM
- 使用异步刷新避免缓存击穿
- 监控缓存命中率,动态调整策略

### 7.2 数据库优化

#### 7.2.1 索引优化
- 高频查询字段建立索引
- 组合索引优化
- 定期分析慢查询

#### 7.2.2 分库分表
- 按时间分表: issue_202604
- 按租户分库: db_tenant_001

### 7.3 向量检索优化

#### 7.3.1 索引选择
- 小数据量(<100万): FLAT
- 中数据量(100万-1000万): IVF_FLAT
- 大数据量(>1000万): HNSW

#### 7.3.2 检索优化
- 元数据预过滤
- 向量分区
- 批量检索

### 7.4 异步处理

#### 7.4.1 消息队列
- 文件处理: 异步队列
- 通知发送: 异步队列
- 统计计算: 延迟队列

#### 7.4.2 线程池

@Configuration
