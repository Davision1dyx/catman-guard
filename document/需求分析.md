# 智能运维Agent系统需求分析

## 1. 需求概述

智能运维Agent系统旨在通过AI技术提升运维效率,构建"问答-知识-管理"三位一体的运维支持平台。系统核心围绕三大模块展开:

1. **智能运维模块**: 提供基础问答、内部知识问答、运维问答三类服务
2. **文件管理模块**: 为知识库提供文件上传、处理、向量化能力
3. **管理后台模块**: 提供统计分析、问题管理、人员管理等功能

## 2. 业务流程分析

### 2.1 核心业务流程

#### 2.1.1 基础问答流程

```
用户提问 → 加载会话历史 → LLM推理 → 流式输出 → 保存会话
```

**关键点**:
- 需要维护会话上下文,保证多轮对话连贯性
- 采用流式输出,提升用户体验
- 会话历史存储,支持会话恢复和历史查询

**参考实现分析**:
catman 项目中的实现要点:
- 使用 `ChatMemory` 管理会话历史
- 使用 `Sinks.Many` 实现流式输出
- 使用 `ContextSessionService` 持久化会话数据
- 支持会话中断与恢复

#### 2.1.2 内部知识问答流程

```
用户提问 → 向量检索 → 相关文档片段 → 构建Prompt → LLM生成答案 → 输出答案+来源
```

**关键点**:
- 向量检索的准确性和速度直接影响用户体验
- 需要合理设置相似度阈值和返回文档数量
- Prompt设计需要平衡准确性和回答质量
- 需要显示知识来源,增强可信度

**参考实现分析**:
WZAgent/rag 模块的实现要点:
- 使用 `VectorStore` 进行向量存储和检索
- 使用 `QuestionAnswerAdvisor` 自动完成检索增强
- 支持元数据过滤(如按文件名、版本等)
- 支持多种向量数据库

#### 2.1.3 运维问答流程

```
用户提问 → 意图识别 → 分支处理:
├─ 简单问题 → 直接回答
└─ 复杂问题 → 引导填写问题单 → 提交问题单 → 自动派发 → 跟踪处理
```

**关键点**:
- 意图识别的准确性决定后续流程
- 问题单模板需要覆盖运维场景
- 自动派发需要考虑人员技能和负载
- 问题跟踪需要状态机管理

**意图分类**:
1. **直接回答**: 简单的配置查询、概念解释、操作指导
2. **问题单**: 复杂故障、需要人工介入、需要现场处理
3. **知识库查询**: 需要查询历史案例、文档资料

**问题单字段设计**:
```json
{
  "问题ID": "自动生成",
  "问题类型": "故障|咨询|需求|其他",
  "项目名称": "关联项目",

  "问题描述": "详细描述",
  "影响范围": "严重|一般|轻微",

    "软件版本": "v1.2.3",

  "更新时间": "2026-04-27 11:00:00"

```

#### 2.1.4 文件处理流程

```
文件上传 → 格式校验 → 文本提取 → 文本清洗 → 文本分片 → 向量化 → 存储
```

**关键点**:
- 支持多种文件格式,需要不同的解析器
- 文本清洗影响向量质量
- 分片策略影响检索效果
- 需要支持增量处理

**文件处理策略**:
1. **文本提取**:
    - PDF: 使用 Apache PDFBox 或 Tika
    - Word: 使用 Apache POI
    - Markdown: 纯文本处理
    - Excel: 按行读取

2. **文本清洗**:
    - 去除特殊字符
    - 去除多余空白
    - 统一编码格式

3. **文本分片**:
    - 按 Token 数量分片(推荐 512-1024 tokens)
    - 保留重叠部分(10-20%),维持上下文
    - 按段落边界分片,避免语义断裂

4. **向量化**:
    - 选择合适的 Embedding 模型
    - 批量处理提升效率
    - 异步处理避免阻塞

### 2.2 问题单生命周期

```
[新建] 
  ↓ 用户提交问题单
[待处理] ←─────┐
  ↓ 值班人员接收      │
[处理中]          │ 转派
  ↓ 处理完成          │
[已解决] ─────────┘
  ↓ 确认关闭
[已关闭]
```

**状态说明**:
- **新建**: 用户刚提交,未分配处理人
- **待处理**: 已分配处理人,等待处理
- **处理中**: 处理人正在处理
- **已解决**: 问题已解决,等待用户确认
- **已关闭**: 用户确认或自动关闭

**状态转换规则**:
- 新建 → 待处理: 自动派发或手动分配
- 待处理 → 处理中: 处理人接手
- 处理中 → 已解决: 提交解决方案
- 处理中 → 待处理: 转派给其他人
- 已解决 → 已关闭: 用户确认或超时自动关闭

### 2.3 知识库更新流程

```
问题单解决 → 管理人员审核 → 问题向量化 → 加入知识库 → 更新向量索引
```

**关键点**:
- 需要人工审核,确保知识质量
- 提取问题和解决方案作为知识对
- 向量化后加入运维问答知识库
- 需要版本管理,支持回滚

### 5.3 关键技术方案

#### 5.3.1 流式输出方案

**问题**: AI回答生成需要时间,长时间等待影响用户体验

**方案**: 采用 Server-Sent Events (SSE) 实现流式输出

**参考实现**:
```java
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<String> streamChat(@RequestParam String message) {
    return chatClient.prompt()
        .user(message)
        .stream()
        .content();
}
```

**前端接收**:
```javascript
const eventSource = new EventSource('/stream?message=xxx');
eventSource.onmessage = (event) => {
    // 实时显示回答
    appendMessage(event.data);
};
```

#### 5.3.2 会话记忆方案

**问题**: 多轮对话需要保持上下文

**方案**: 使用 ChatMemory + 数据库持久化

**参考实现**:
```java
// 使用 InMemoryChatMemory 或自定义持久化
ChatMemory chatMemory = new InMemoryChatMemory();
chatMemory.add(conversationId, new UserMessage(question));
chatMemory.add(conversationId, new AssistantMessage(answer));

// 持久化到数据库
contextSessionService.saveMessage(conversationId, question, answer);
```

#### 5.3.3 向量检索方案

**问题**: 如何从大量文档中快速检索相关内容

**方案**: 向量数据库 + 相似度检索

**参考实现**:
```java
SearchRequest searchRequest = SearchRequest.builder()
    .query(userQuestion)
    .topK(5)
    .similarityThreshold(0.7)
    .filterExpression("version == 'v1.2.3'")
    .build();

List<Document> documents = vectorStore.similaritySearch(searchRequest);
```

**优化策略**:
1. **元数据过滤**: 先按版本、分类等条件过滤,再进行向量检索
2. **混合检索**: 结合关键词检索和向量检索
3. **重排序**: 对检索结果进行二次排序,提升准确性

#### 5.3.4 意图识别方案

**问题**: 如何准确识别用户问题类型

**方案**: Few-shot Prompting + 结构化输出

**实现思路**:

#### 5.3.5 文件处理方案

**问题**: 多种文件格式需要统一处理

**方案**: 策略模式 + 异步处理

**实现思路**:
```java
// 文件读取策略
public interface DocumentReaderStrategy {
    List<Document> read(File file);
}

// PDF策略
public class PDFReaderStrategy implements DocumentReaderStrategy {
    public List<Document> read(File file) {
        // 使用PDFBox提取文本
    }
}

// 文件处理Handler
public class DocumentReaderHandler {
    private Map<String, DocumentReaderStrategy> strategies;
    
    public List<Document> handle(File file) {
        String fileType = getFileType(file);
        DocumentReaderStrategy strategy = strategies.get(fileType);
        return strategy.read(file);
    }
}
```

## 7. 非功能性需求分析

### 7.1 性能分析

#### 瓶颈点
1. **LLM调用延迟**: 首字延迟 1-3秒,完整回答 5-30秒
2. **向量检索**: 百万级文档检索延迟 100-500ms
3. **文件处理**: 大文件(>10MB)处理耗时较长

#### 优化方案
1. **流式输出**: 降低感知延迟
2. **缓存热点问题**: 减少LLM调用
3. **向量索引优化**: 使用HNSW或IVF索引
4. **异步处理**: 文件处理放入消息队列
5. **CDN加速**: 静态资源缓存

### 7.2 可扩展性分析

#### 水平扩展点
1. **应用服务**: 无状态设计,可横向扩展
2. **数据库**: 主从分离,读写分离
3. **向量数据库**: 分片和副本
4. **缓存**: Caffeine (本地缓存,单机或小规模集群)

#### 垂直扩展点
1. **LLM**: 更强的模型或硬件
2. **向量维度**: 更高维度的向量表示

### 7.3 安全性分析

#### 安全威胁
1. **Prompt注入**: 用户输入恶意Prompt
2. **数据泄露**: 未授权访问敏感数据
3. **会话劫持**: 窃取会话Token
4. **DDoS攻击**: 大量请求耗尽资源

#### 防护措施
1. **输入过滤**: 对用户输入进行验证和清洗
2. **权限控制**: RBAC + 数据权限
3. **会话管理**: JWT + Refresh Token + HTTPS
4. **限流熔断**: 防止恶意请求
5. **审计日志**: 记录所有敏感操作

### 7.4 可用性分析

#### 故障点
1. **LLM服务不可用**: OpenAI API故障
2. **数据库故障**: MySQL宕机
3. **向量库故障**: Milvus不可用
4. **网络故障**: 服务间通信失败

#### 容错方案
1. **多LLM提供商**: OpenAI、Azure、本地模型互备
2. **数据库高可用**: 主从切换
3. **向量库集群**: 多副本
4. **熔断降级**: 核心功能降级为普通问答
5. **重试机制**: 自动重试失败请求
